Sensitivity Analysis — 3-Region Curve

Crawley Ch 15 §15.5 · Fig 15.13 · "Three regions: Region I (insensitive, near 0%), Region II (sensitive middle, S-curve), Region III (insensitive, near 100%)" · default operating point in Region II = NOT ROBUST

Crawley Fig 15.13 (viewed directly) — % homogeneous architectures on Pareto front vs dissimilar-component penalty · dashed vertical boundaries · line 7208
Robustness rule (Crawley §15.5): A result in Region II is NOT robust. "For a connection weight of only 0.02, the percentage is already below 30% — fully cross-strapped result not robust to even modest parameter changes." For c1v: if sensitivity analysis places the recommended architecture in Region II, set robustness_score < 0.8 and trigger fuzzy_pareto with euclidean_threshold. Surface an open question via chat bridge before committing. Do not claim a recommendation "proven" until the operating point is in Region I or III.
Book Fig 15.13 (viewed directly): Y = "% of homogeneous architectures on the PF", X = "Dissimilar component penalty/maximum connection weight" from -2.0 to +2.0. S-curve stepping from ~3% to ~97%. Region I (DCP < -0.5): flat at 0% — no homogeneous archs on PF. Region II (-0.5 to +1.1): steep S-curve, each tick of X changes result by 10–20%. Region III (DCP > +1.1): flat at 100% — all Pareto archs homogeneous. Dashed vertical lines at -0.5 and +1.1 exactly. Default DCP=0 sits in Region II. Book Fig 15.12: Same 3-region structure for connection weight — % fully cross-strapped on PF drops from 100% to <30% with connection_weight increase of 0.02.

Optimization Patterns (Ch 16 Table 16.2)

Crawley Ch 16 §16.4 · "Six Patterns appear over and over when system architecture optimization problems are formulated" · NEOSS (NASA Earth Observing Satellite System) as running example

Crawley Table 16.2 (6 patterns) + Fig 16.3 (DECISION-OPTION pictorial) + Fig 16.5 (DOWN-SELECTING) + Figs 16.8–16.9 (ASSIGNING styles) · pp. 373–420
DECISION-OPTION
A group of decisions, each with its own discrete set of options. Most general Pattern. Best for heterogeneous, independent choices (Apollo 9 decisions). Can represent ANY programmed decision problem.
Tradespace = Πi |Oi| (product of options per decision)
Representation: morphological matrix · Tasks 3, 4 · Book Fig 16.3: two architectures A=[2,1,3] and A=[1,1,2] shown side-by-side, dashed boxes=unselected, solid boxes=selected
DOWN-SELECTING
Choose a subset from a set of m candidate elements. Binary decisions (include/exclude). Accounts for synergies, interferences, redundancies between elements. NEOSS: instrument selection from 8 candidates.
Tradespace = 2m · Extends the 0/1 knapsack problem
Heuristic: single-point crossover in genetic algorithms keeps good element combinations together · Book Fig 16.5: 8 elements, left=5 selected (black solid), right=3 selected
ASSIGNING
Given a left set and right set of entities, assign each left element to any subset of right elements. NEOSS: assign instruments to orbits (each instrument can go on multiple orbits).
Tradespace = 2mn · Binary m×n matrix
Styles: channelized (each left→exactly one right, Saturn V stages) vs fully cross-strapped (every left→every right, Space Shuttle avionics). Cost vs reliability tradeoff.
PARTITIONING
Partition a set of N elements into non-empty, mutually exclusive, exhaustive subsets. NEOSS: partition instruments into satellites. NP-hard — grows via Bell numbers.
Tradespace = Bell(N) · Bell(5)=52 · Bell(10)≈115,975
Styles: monolithic (Envisat: 10 instruments on 1 satellite) vs fully distributed (1 instrument per dedicated satellite). Evolvability tradeoff.
PERMUTING
Total ordering/sequencing of N elements (bijection to integers 1..N). NEOSS: satellite launch schedule — order in which satellites/missions are launched.
Tradespace = N! · 5!=120 · 10!=3,628,800
Styles: greedy deployment (highest-value elements first, long lead time, high programmatic risk) vs incremental deployment (deliver value ASAP, build up capability).
CONNECTING
Given a set of nodes, choose which graph edges to include. GNC sensor-computer connectivity. Properties: latency, throughput, reliability, scalability.
Undirected: 2m(m-1)/2 · Directed: 2m²
Topologies: bus, star, ring, mesh, tree, hybrid. Figs 16.8–16.9: channelized (one sensor→one computer) vs fully cross-strapped (all sensors→all computers).
NEOSS multi-pattern composition (Book §16.4 / Fig 16.15): NEOSS requires all three patterns together — DOWN-SELECTING (instrument selection) + ASSIGNING (instrument → orbit packaging) + PERMUTING (launch schedule). These sub-problems are coupled: "One cannot make an optimal decision in the instrument selection problem without thinking about the packaging of the selected instruments." c1v's compositionSchema captures this via operator: cartesian_product | sequential | hierarchical | shared_resource. If sub_problems.length > 1 without composition, schema rejects with superRefine error.
Book Table 16.2 (lines 7523–7532): Exact pattern list: DECISION-OPTION | DOWN-SELECTING | ASSIGNING | PARTITIONING | PERMUTING | CONNECTING. Each has a short 1-sentence description.
