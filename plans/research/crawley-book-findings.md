## §3 Key Concepts + Formulas

### 3.1 Form / Function / Concept Triad — Crawley's exact formulation

Direct Crawley wording (MIT ESD.34 lecture 3; carried into the 2015 book, Chapters 4–8):

> **"A concept is a product or system vision, idea, notion or mental image which maps Function to Form. It embodies the principle of operation and includes an abstraction of form. Concept is not a product/system attribute — it is a mapping."**

- **Form** = the physical/informational embodiment (Chapter 4). In software: the code structures, data structures, deployed runtime artifacts. Analyzed via: elements → structure → formal relationships → formal context.
- **Function** = what the system *does* — process acting on an operand to produce value (Chapter 5). Split into: **external function** (value delivery to beneficiaries) + **internal function** (instrument functions + functional interactions). Operands are central — value is created by a change of state of an operand.
- **Concept** = the *mapping* from Function → Form. It carries the principle of operation and an abstraction of form. The architecture is produced by **expanding the concept** (Chapter 8): expansion to functional architecture, then expansion to form.
- **System Architecture** (Chapter 6) = the synthesis of form and function — "the embodiment of concept, the allocation of function to form, and the definition of relationships (between elements, and to context)."

### 3.2 Decision math (Chapter 14 + 15 + 16)

From the freely-available Chapter 14 PDF and the O'Reilly-indexed section titles:

**Decision representation:**
- **Decision tree** — used for simple sequential decisions, limited in simulation.
- **Decision network** — "a more general version of decision trees where the tree-structure condition is relaxed to allow arbitrary topologies between decision nodes, chance nodes, and leaf nodes" (Ch 14). This is the exact object c1v's M4 decision-matrix schemas should implement.
- **Design Structure Matrix (DSM) of decisions** — N×N matrix where row/col = one of the N decisions; entries indicate connections. Connection types: (a) logical constraints, (b) "reasonableness" constraints, (c) connections through shared metrics. (Ch 14)

**The Four Tasks of Decision Support (Ch 14)** — the canonical decomposition c1v's decision pipeline should implement:
1. Represent the decisions (tree / network / DSM).
2. Evaluate each candidate (metrics / utility / cost / risk).
3. Search / optimize the decision space.
4. Communicate + explain the chosen architecture.

**Tradespace exploration (Ch 15):**
- **Tradespace plot** — 2D (or higher) scatter of metrics (e.g., IMLEO vs. probability-of-mission-success); each point = a logically feasible combination of decision assignments.
- **Pareto Frontier** — the set of points that are best in a given metric with all other metrics held fixed.
- **Structure of the tradespace** (15.4) — holes, subgroups, fronts arise from: discrete metrics, different dynamic ranges, physical laws. Crawley explicitly warns: "Pareto analysis is important but not sufficient — look at the structure of the tradespace as a whole."
- **Sensitivity Analysis** (15.5) — how much does the metric change when a decision assignment flips? Governs the priority order in which decisions should be fixed.
- **Organizing Architectural Decisions** (15.6) — structure the N decisions by priority (sensitivity) and by DSM connectivity; make the highest-sensitivity, least-connected decisions first.

**Optimization (Ch 16):**
- Formulation as a constrained mixed-integer / combinatorial optimization over the decision vector.
- Patterns: down-selecting, assigning, partitioning, permuting, connecting, permuting-with-substitution. (The "six patterns" are from Selva's 2016 *Systems Engineering* journal paper "Patterns in System Architecture Decisions" — folded into Chapter 16.)
- Solvers covered: full factorial, heuristic search, evolutionary algorithms (NSGA-II-style), knowledge-based rule systems.
- NEOSS (NASA Earth Observing Satellite System) is the running large-scale case study.

### 3.3 Concept-quality metric — explicit formal score?

**Finding:** Crawley does **not** define a single scalar "concept-quality" formula in the book. Quality of a concept is judged **heuristically and multi-dimensionally** via the following evaluation axes (Chapters 7, 8, 11, 12 — Cornell CESYS525 lecturers have codified these as a rubric):

1. **Solution-neutrality** — does the concept express function without presupposing form? (Ch 7)
2. **Completeness** — does it map all primary value-delivering functions? (Ch 8)
3. **Holism / integrated concept** — do concept fragments combine into one coherent principle of operation rather than a list? (Ch 8)
4. **Traceability to needs/goals** — does it satisfy the goals from Ch 11? (Ch 11 — criteria for goals)
5. **Generativity** — does it yield multiple downstream architectures (is it a concept or a single design)? (Ch 12)
6. **Robustness to ambiguity** — does it survive the Ch 9 ambiguity / architect-role analysis?

**For c1v:** a concept-quality score can be *constructed* as a weighted sum of these six axes (each 0–1), but the scalar formula is c1v's own — not Crawley's. Mark it in `.planning/` notes as "derived from Crawley Ch 7/8/11/12 heuristics, not a verbatim textbook formula."

### 3.4 OPM (Object-Process Methodology) integration

- Chapter 3 (§3.5 "Architecture Representation Tools: SysML and OPM") introduces OPM as a first-class representation language alongside SysML.
- Chapters 4–8 use OPM-style process/object diagrams to illustrate function→form mappings. The operand-based function definition in Ch 5 is structurally OPM (Dori's processes operate on objects, transforming their state — identical to Crawley's "process acts on operand to create value").
- **Crawley collaborated with Dov Dori** on the paper "OPM-Based System Function and Architecture: Definitions and Operational Templates" (Soderborg, Crawley, Dori — freely available on academia.edu). This paper is the canonical bridge between Crawley's F/F/C triad and Dori's OPM and is cited in the book.
- Crawley wrote the foreword to Dori's "Model-Based Systems Engineering with OPM and SysML."
- MIT ESD.34 (IAP 2007, Crawley) has a dedicated OPM tutorial PDF ([lec2_opm.pdf](https://ocw.mit.edu/courses/esd-34-system-architecture-january-iap-2007/997dd6ff90ce2edcf9a451d6aabe5585_lec2_opm.pdf)) — identical OPM framing as the book.

**For c1v:** OPM is the natural notation for Module 5 form-function mapping. The operand-centric view aligns cleanly with the FFBD work in M3.

### Core concept
Ch 1 establishes two load-bearing definitions: (a) **architecture** = "an abstract description of the entities of a system and the relationship between those entities" (line 399); in systems built by humans, "this architecture can be represented as a set of decisions." (b) The **central assertion** of the book: "early decisions can be analyzed and treated. … structured creativity is better than unstructured creativity" (line 409). ### Key heuristics / operators / formulas
- **The first-10-decisions rule** (line 606, set up in Ch 1 and formalized in Ch 14): the first ~10 architectural decisions determine the majority of performance and cost.
- **Good-architecture checklist** (line 389): well-architected systems "meet stakeholder needs and deliver value," "integrate easily, evolve flexibly, and operate simply and reliably."

### Patcher-ready seed (200–500 words)
System architecture, per Crawley Ch 1, is "an abstract description of the entities of a system and the relationship between those entities," and "in systems built by humans, this architecture can be represented as a set of decisions." KB-1 must therefore scope a project as a *decision object* from the first interaction — the c1v intake phase is architecturally identical to what Crawley calls "driving ambiguity from the upstream process" (Ch 9). The advantages of good architecture that Ch 1 enumerates are: the system meets stakeholder needs and delivers value; it integrates easily, evolves flexibly, and operates simply and reliably. Ch 1's worked failure case is NPOESS (lines 413–423): a 1994 merged weather-satellite program that absorbed every instrument capability from both parent programs without re-deriving a coherent system concept, trapping architectural performance "in an unreachable corner of its envelope" (VIIRS was to do three instruments' work at less mass/volume than one). ## Ch 2 — System Thinking [FILLED 2026-04-24]

**Source:** markdown lines 493–1050 / pdf pages ~22–48.
**Target KB:** 1-defining-scope.
**Existing in-tree work:** none.

### Core concept
System thinking is "reasoning about a question, circumstance, or problem explicitly as a system — a set of interrelated entities" (line 495). Ch 2 defines the four-task system-thinking procedure plus the cornerstone concept of **emergence**.

### Key heuristics / operators / formulas
- **Definition: System** (Box 2.1, line 520): "A system is a set of entities and their relationships, whose functionality is greater than the sum of the individual entities." Two tests: (1) entities that interact, (2) emergent function.
- **Principle of Emergence** (Box 2.2, line 576): "A system is not the sum of its parts, but the product of the interactions of those parts." Change propagates unpredictably; failure = desired emergence doesn't appear, or undesired emergence appears.
- **Emergence 2×2 taxonomy** (Table 2.1, lines 541–547): anticipated/unanticipated × desirable/undesirable (car transports people / car provides freedom / car burns hydrocarbons / car kills people).
- **Form-Function definition pair** (line 639): Form = what a system IS (physical/informational embodiment, static, instrumental). Function = what a system DOES (process + operand). Function consists of *process* (the action/transformation) acting on an *operand* (the object whose state changes).

The worked transportation example: beneficiary=traveler, need=visit-client-in-another-city, SN operand=traveler, benefit attribute=location, other attributes=alone+light-luggage, SN process=changing-location (transporting), process attributes=safely + on-demand. The summary SNF is *"safely and on demand transporting to a new location a traveler with light luggage"* — NOT "fly the traveler" (flying is a specific process, not SN). Concept (§7.3, Box 7.2) is the mapping from SNF to form: specialize the SN operand + SN process to concrete ones, add a specific instrument + attributes. Box 7.3 enumerates the six operand-specialization patterns (completely different / process preserved / part-of / type-of / attribute-of / information-about) and three process-specialization patterns (different / type-of / attribute-added). For rich concepts (§7.4), unpack the concept into internal functions (e.g., transporting = lifting + propelling + guiding) and build a **morphological matrix** — rows = internal functions, columns = candidate instruments. Pick one instrument per row; the column choice yields a named integrated concept. Wheels-wheels-wheels = car; wheels-wheels-ground = train; wings-propeller-rudder = propeller-aircraft; propeller-propeller-propeller = helicopter. SNFs exist in a **hierarchy** (§7.3): at the next level up, today's specific function becomes tomorrow's SN intent (deal-closing ← client-preference-learning ← client-meeting ← traveling ← flying). The architect should understand two levels of this hierarchy before committing to a concept. For c1v KB-5, the SNF procedure is how intake phase refuses to prematurely over-specify form, and the concept template (Figure 7.4) is the schema that KB-5 outputs.

## Ch 13 — Decomposition as a Tool for Managing Complexity [FILLED 2026-04-24]

**Source:** markdown lines 5992–6569 / pdf pages ~300–322.
**Target KB:** 2-dev-sys-reqs (verify-only) AND 3-ffbd-llm-kb.
**Existing in-tree work:** `crawley-ch13-decomposition-heuristics.md` (456 lines) in `apps/product-helper/.planning/phases/13-Knowledge-banks-deepened/2-dev-sys-reqs-for-kb-llm-software/`. **Cross-reference that file first.** If the existing file does not include the 12 planes of Box 13.8, this section supplies them.

### Core concept
Complexity is "many interrelated, interconnected, or interwoven elements and interfaces" (line 6010) — an absolute, measurable property. Decomposition is "an active choice made by the architect" (Box 13.5, line 6206) — the highest-leverage decision in architecting. The goodness of a decomposition is evaluated by the **2-Down-1-Up principle** (Box 13.6, line 6231).

### Key heuristics / operators / formulas
- **Complexity measures** (Box 13.1, line 6020): N1 = #things, N2 = #types of things, N3 = #interfaces, N4 = #types of interfaces. Simple additive measure: C = N1 + N2 + N3 + N4. Boothroyd-Dewhurst variant: C = (N1 × N2 × N3)^(1/3).
- **Three complexity categories**: (a) **essential complexity** (Box 13.3, line 6115) — the minimum required to deliver the specified functionality within a given concept; (b) **actual complexity** — the real measured complexity (always ≥ essential, due to decomposition/abstraction overhead); (c) **gratuitous complexity** — actual minus essential. Avoid gratuitous complexity.
- **Complex vs. complicated** (§13.2, line 6063): complex is about the system; complicated is about the observer's ability to perceive. "Build systems of the necessary level of complexity that are not complicated" (line 6089).
- **Principle of Apparent Complexity** (Box 13.2, line 6066): decomposition, abstraction, and hierarchy reduce apparent complexity but may increase actual complexity (each decomposition adds an interface — see Figure 13.8).
- **Principle of the 2nd Law** (Box 13.4, line 6175): "Actual complexity always exceeds essential complexity" — there is creep due to abstractions/decompositions/hierarchy; manage it.
- **Principle of Decomposition** (Box 13.5, line 6206): decomposition choices are (1) how many internal boundaries, (2) where to place them, (3) **in what plane** to consider them (the plane is the hardest choice).
- **Principle of 2-Down-1-Up** (Box 13.6, line 6231): "The goodness of a decomposition at Level 1 cannot be evaluated until the next level down has been populated and the relationships identified (Level 2). Select the modularization at Level 1 that best reflects the clustering of relationships among the elements at Level 2." The algorithmic instantiation is Ch 8 Box 8.1.
- **Box 13.8 — 12 Planes of Decomposition** (line 6312): (1) **delivered function and emergence** — don't spread externally-delivered functions across many elements (hard to integrate); (2) **form and structure** — cluster high-connectivity form, don't interface across high-connection regions; (3) **design latitude and change propagation** — group tightly-coupled designs; (4) **changeability and evolution** — place interfaces so modules combine into platforms; (5) **integration transparency** — interfaces should support testing and visibility; (6) **suppliers** — interface where suppliers can work independently; (7) **openness** — interfaces open to third parties balance innovation vs. IP leakage; (8) **legacy components** — reuse constrains decomposition; (9) **clockspeed of technology change** — interfaces should allow different-rate technology swap-out; (10) **marketing and sales** — enable differentiating features without architectural change; (11) **operations and interoperability** — decompose to expose wear parts and operator touch points; (12) **timing of investment** — phase development spending; (13) **organization** — match modularization to organization (Conway's Law).
- **Principle of Elegance** (Box 13.7, line 6280): "Elegance is appreciated internally by the architect when a system has a concept with low essential complexity and a decomposition that aligns many of the planes of decomposition simultaneously."
- **Conway's Law** (line 6306): "organizations which design systems are constrained to produce designs which are copies of the communication structures of these organizations." Confirmed with recent empirical evidence on software product modularity.

## Ch 16 — Formulating and Solving System Architecture Optimization Problems [MINOR-PATCH 2026-04-24]

**Source:** markdown lines 7394–8146 / pdf pages ~373–420.
**Target KB:** 4-decision-net AND 6-hoq (§Patterns).
**Status:** OK in prior audit. This patch adds one-sentence definitions of the 6 Patterns (the nice-to-have flagged in the gap audit).

### The 6 Patterns — one-sentence definitions (per Table 16.2, line 7523)

1. **DECISION-OPTION** (line 7538) — A group of decisions where each decision has its own discrete set of options. Tradespace size = product of options per decision. Most general Pattern; use it when decisions are heterogeneous (Apollo 9 decisions).
2. **DOWN-SELECTING** (line 7584) — A group of binary decisions representing a subset chosen from a set of candidate entities. Tradespace size = 2^m. Captures synergies/interferences/redundancies between elements (NEOSS instrument selection). Extends the 0/1 knapsack problem by modeling element-to-element interactions.
3. **ASSIGNING** (line 7636) — Given two sets (left and right), assign each left element to any subset of right elements. Tradespace size = 2^(mn). Two extreme styles: **channelized** (each left element → exactly one right element — e.g., Saturn V stages) vs. **fully cross-strapped** (every left → every right — e.g., Space Shuttle avionics, "Total Football"). Trade-off = cost vs. reliability/throughput (Box 16.2).
4. **PARTITIONING** (line 7722) — Partition a single set of N elements into non-empty disjoint subsets that are mutually exclusive and exhaustive. Tradespace size grows via Bell numbers (52 partitions for N=5, 115K for N=10). Two extreme styles: **monolithic** (all in one subset — Envisat with 10 instruments) vs. **fully distributed** (each in its own subset — dedicated single-instrument satellites). Trade-off = synergies/interferences + redundancy + evolvability (Box 16.3).
5. **PERMUTING** (line 7796) — Bijection between a set of N elements and the integers 1..N (equivalently: a total ordering or sequencing). Tradespace size = N! Two extreme styles: **greedy deployment** (highest-value elements first, long lead time, high programmatic risk) vs. **incremental deployment** (deliver some value ASAP, then build up). Trade-off = delivery risk vs. total cost (Box 16.4).
6. **CONNECTING** (line 7841) — Given a set of nodes, choose a graph (set of edges). Tradespace size = 2^(m(m-1)/2) for undirected/no-self, 2^(m²) for directed/self-connected. Styles = network topologies (bus, star, ring, mesh, tree, hybrid). Relevant properties: latency, throughput, reliability, scalability (Box 16.5).

The six Patterns support Tasks 1–6 of Table 16.1 (decomposing, mapping, specializing, characterizing, connecting, selecting-goals) but not one-to-one — most problems are formulatable in multiple Patterns; pick the one that exposes the most structure for the optimizer.
