# Crawley Ch 1 — Introduction to System Architecture

System architecture, per Crawley Ch 1, is "an abstract description of the entities of a system and the relationship between those entities," and "in systems built by humans, this architecture can be represented as a set of decisions." KB-1 must therefore scope a project as a *decision object* from the first interaction — the c1v intake phase is architecturally identical to what Crawley calls "driving ambiguity from the upstream process" (Ch 9).

The advantages of good architecture that Ch 1 enumerates are: the system meets stakeholder needs and delivers value; it integrates easily, evolves flexibly, and operates simply and reliably.

Ch 1's worked failure case is NPOESS (lines 413–423): a 1994 merged weather-satellite program that absorbed every instrument capability from both parent programs without re-deriving a coherent system concept, trapping architectural performance "in an unreachable corner of its envelope" (VIIRS was to do three instruments' work at less mass/volume than one). The program was canceled in 2010 at $8.5B over its $6.5B baseline.

Ch 1's paired success heuristic, from the 1962 Apollo Lunar Orbit Rendezvous decision (line 379), is that a *single early architectural decision*, made before the majority of staff is hired and before contracts are awarded, can "eliminate many possible designs" and give design teams a stable starting point.

KB-1 scoping therefore has two explicit deliverables: (1) identify the one-or-few early decisions that carry this level of leverage, and (2) refuse to treat architecture as downstream-of-design.

Ch 1 also draws the book's own "principles / methods / tools" distinction (line 450): principles are enduring truths, methods are organized procedures, tools are contemporary facilitators — a labeling convention KB-1 should adopt verbatim for its own content, so every c1v utterance can be tagged with its epistemic status.

Ch 1 closes by framing the four parts of the book as the canonical pipeline: Part 1 = system thinking; Part 2 = analysis (reverse-engineer existing architectures); Part 3 = synthesis (create new architectures); Part 4 = architecture as decisions.
