The LinkedIn Generative AI Application Tech Stack: Personalization with Cognitive Memory Agent

Authored byPraveen Kumar Bodigutla

March 26, 2026

Co-authors: Co-authored byPraveen Kumar Bodigutla, Co-authored byKarthik Ramgopal, Co-authored byXiaofeng Wang, Co-authored byShangjin Zhang, and Co-authored byZhentao Xu

In 2025, we announced the global availability of Hiring Assistant in English, LinkedIn’s first AI agent for recruiters, and shared a behind-the-scenes look at how our generative AI application stack evolved to power agents. Through that work, we learned that delivering a truly great agentic experience required more than a capable model. Our agent needed strong domain intelligence grounded in a rich organizational context and an ability to get better over time. This required memory to learn from interactions and support an increasingly personalized, adaptive, and useful experience for recruiters. 

To address these needs, we built the Cognitive Memory Agent (CMA), a horizontal memory platform to build stateful and context-aware AI agents at scale. Traditional memory systems require users to explicitly specify what they want an agent to remember. By contrast, CMA intelligently stores and retrieves contextually relevant information. This is done across multiple memory stores that provide different levels of knowledge that enable deep personalization in agents like Hiring Assistant. 

In this blog, we’ll share insights about the CMA architecture by diving into different layers that compose our memory platform, the approach to store and retrieve data safely, and the practical application of CMA within the LinkedIn Hiring Assistant. 

The Cognitive Memory Agent architecture

CMA allows application agents, such as our Hiring Assistant, to move beyond a simple recall of previous interactions. Below the surface, it enables continuity beyond context windows, runtime learning “on the job,” and makes tool usage compositional. Agents can fetch previous interactions and preferences, resolve conflicts between them, and avoid redundant planning and retrieval steps.

In multi-agent systems, multiple agents can use CMA-backed memory as a coordination substrate. A kind of durable scratchpad that prevents conflicting actions and enables structured collaboration.

However, introducing memory increases system complexity. Once a state persists across steps, a well-architected memory system must handle consistency, latency, storage and access patterns, isolation, privacy, compaction, and observability. Memory stops being incidental context and becomes a first-class primitive with explicit read/write semantics and lifecycle management.

At the high level, CMA tackles the memory challenge through three core components:

  • Memory layers consisting of conversational, episodic, semantic and procedural memory. Each memory layer is exposed via tool abstractions.
  • An ingestion layer that determines how to interpret unstructured inputs, what information to extract, and when and how to store it.
  • A retrieval orchestration layer that infers user intent expressed in natural language, dynamically retrieves relevant memories across layers, and synthesizes responses. Instead of merely being an abstraction for simple recall using embedding-based retrieval, the retrieval layer is optimized for quality and efficiency. It also uses reasoning and planning over memory tools to generate contextually relevant responses.

To better understand the CMA architecture, we’ll start by exploring the various types of memory that contribute to the platform and application agent capabilities.

Memory layers

CMA maintains multiple types of memory, each with different semantics, latency characteristics, and abstraction levels. Conversational memory captures a compressed in-session state. Then we consider various types of memory that we view as sub-categories of long-term memory. That consists of: Episodic memory which stores time-indexed activity traces; Semantic memory that aggregates structured knowledge and inferred preferences; Procedural memory which captures invocation plans and reasoning traces. 

Conversational memory layer

Conversational memory stores, indexes, and retrieves prior turns in a dialogue between the user, application and memory agent so that future turns can incorporate relevant history without exceeding context limits. Messages are exchanged over the highly-scalable and fault tolerant LinkedIn messaging platform. In addition to storing conversation raw text, indexed by time-stamp, computed embeddings for the conversations are persisted in the vector store. This creates two parallel representations: a chronological log and a semantic index.

During retrieval, most recent interactions and similar conversations (fetched via Semantic retrieval) are available for CMA to generate contextually relevant responses for a given natural language query. Past dialogues are summarized periodically to achieve context compression.

Long-term memory layer

Beyond conversational memory lies long-term memory. It is the layer that allows an agent to accumulate durable knowledge about a user, or the entities/environment they interact with across sessions. 

Our initial approach to solving this problem was to build experiential memory, which offered a durable hierarchical KV store, built atop Couchbase and Espresso, that applications could use to store and retrieve preferences. The limitation was that this was just systems infrastructure with no intelligence. Each application had to handle preference extraction, determine how to index it, and then use the provided read/write APIs for storage and retrieval. This made every integration custom-built and maintained manually, preventing ubiquitous adoption.

As we developed CMA, we enhanced long-term memory to dynamically extract and synthesize implicit preferences. Rather than relying solely on explicit user instructions, like say “I prefer a concise writing style in my InMails,” the system infers preferences by leveraging high-level instructions provided by an application agent developer. The high-level instructions, also known as memory prompts, capture details about the type of task, interaction an application agent supports and preferences that need to be mined and persisted into respective memory stores. Both raw and aggregated preferences are persisted as structured preference records, which are stored separately from conversational memory and are dynamically retrieved by the orchestrator. 

Long-term memory is organized into three sub-categories that mirror cognitive models:

  • Episodic activity memory: Records specific past events or interactions. 
    • Episodic records are timestamped and contextual, allowing the agent to reference activities that are similar and within a specific time range. For example, this layer captures signals like “the recruiter archived a candidate as they lack specific skills in machine learning.” The next time the recruiter tries to hire another machine learning engineer, the agent might use these signals to adjust outreach strategy and content by highlighting the specific skill they are looking for. 
    • Episodic memory builds situational awareness and refines the behavior based on activity signals. The obvious challenge in extracting this memory is correctly identifying episode boundaries, staleness and conflict resolution. Further subtleties include temporal segmentation when events of interest overlap in time or interleave with other tasks, associating the episode with the right context, and capturing just enough details to make the episode useful for the future without overfitting.
  • Aggregated semantic memory: Stores preferences as generalized knowledge derived from repeated interactions of a user. 
    • These preferences are aggregated at different levels of granularity. The challenges in extraction somewhat mirror episodic memory, with the episodes in question here being longer arcs of time. For example, this layer may analyze interactions and capture signals like “The company cannot sponsor visas for this team” or “The company doesn’t hire remote employees.” At retrieval time, the agent may use these signals when drafting a job description for a new job to automatically suggest these details. Semantic memory abstracts away from specific events into general preferences.
  • Procedural memory: Procedural memory influences execution strategy (the“how”) by mining user specific execution patterns from their interactions. 
    • These execution patterns or steps distinctly express the implicit preferences of a user. For example, a recruiter hiring for a “machine learning” role might first filter the suggested candidates by seniority and then select candidates with a specific skill set, before reaching out to them via inMail. On the contrary, a different recruiter accepts the suggested list of candidates for the same role as it is, but applies specific inMail stylistic templates from past outreach. 
    • These steps also translate to choice of specific memory tools and reasoning patterns for the orchestrator. For example, conversational memory is leveraged in the first case to extract latest skills, vs long-term semantic and episodic memory are used in the second case for candidate selection and inMail format respectively. True procedural memory must capture the invariant structure of the workflow while isolating the parameters that change from task to task.

Together, episodic, semantic and procedural memory enable an agent to move from session-level coherence to longitudinal intelligence. Instead of merely remembering what was said, the agent begins to understand how the user works, what has happened before, and what enduring facts define the environment. This enables a high degree of adaptation and personalization. With usage, the agent becomes “smarter” and begins to behave in ways aligned with the user without being reminded each time.

Ingestion

To optimize retrieval latency, we offload data processing to the ingestion phase. Raw user activity and conversation data must be indexed into various memory formats before being stored, enhancing retrieval efficiency. This process involves using LLMs to summarize learnt patterns for long-term memory, extract episodic activities specified in memory prompts and periodically compress conversational memory, all while applying appropriate privacy-preserving techniques. Given the volume of user activity, we perform asynchronous indexing to avoid disrupting online interactions. For this, we offer two pathways: streaming and batch.

Streaming processing handles latency-sensitive tasks, such as conversational memory summarization, preventing overly long conversations that exceed retrieval agents' context limits. It's also used for episodic memory ingestion, which extracts verbalized activity from the raw conversations, ensuring up-to-date user activity storage. 

Batch processing is employed for computation-intensive tasks, like extracting and aggregating semantic memory nodes.

With scope limited to individual user’s accessible memory, our hierarchical semantic memory indexing uses LLM calls to convert their activity data into aggregated Q&A pairs, hierarchical natural language summaries, and facets, that can be later queried by the retrieval layer. Data persistence and retrieval are in conformance with our Responsible AI practices and standard privacy-preserving practices. 

Our approach to hierarchically aggregate information in a tree-like structure, offers several advantages over existing methods like GraphRAG and flattened RAG. By leveraging the natural hierarchy of data, we reduce the need for numerous LLM calls to determine edges, enhancing overall efficiency of memory construction. This also optimizes retrieval by avoiding the need to feed all leaf nodes to the LLM, thus preventing context size limitations.

The tree-like structure allows full parallel processing at each layer, making it well suited for batch workloads. This hierarchical approach simplifies data flow while improving scalability and throughput when operating on large volumes of data. One area where this trait is especially valuable is incremental updates. By processing changes in time-windowed batches, we can smooth out load spikes and propagate updates efficiently in a bottom-up fashion.

Retrieval

Traditional retrieval pipelines treat memory as a static corpus. They embed the query, retrieve the top-k results, append them to the prompt, and let the LLM synthesize an answer. While effective for simple recall, this approach breaks down when memory is layered, heterogeneous, and continuously evolving, as is the case in CMA.

In CMA, retrieval is not a single search operation. It is a reasoning process. We designed the system as a lightweight agentic orchestrator that plans how to access and combine different memory layers before producing an answer. Rather than assuming that all relevant context lives in one store, the orchestrator determines where to look, in what order, and how to reconcile signals across stores.

As described above, CMA maintains multiple types of memory, each with different semantics, latency characteristics, and abstraction levels. A single search operation cannot reliably determine which layer is relevant for a given query, nor can it coordinate interactions across them.

The CMA orchestrator receives the natural language query, relevant memory keys, and the available memory tools. It then generates a retrieval plan. For simple queries, this plan may involve a single semantic lookup. For more complex questions, the orchestrator may sequentially invoke multiple memory providers, refine intermediate queries, and synthesize results across layers before producing a final response.

Consider the question: “Why does a recruiter like this candidate?” Answering this requires more than retrieving a single record. The orchestrator may first consult aggregated semantic memory to understand inferred recruiter preferences and long term hiring patterns. It may then query episodic memory for recent positive feedback on non-identifiable features of similar candidates and qualification refinements (e.g., a previous candidate sent an InMail had skills in machine learning). If the interaction is ongoing, it can incorporate signals from conversational memory. Only after correlating these signals does it synthesize an explanation. Retrieval, in this case, becomes a multi-step reasoning process rather than a one-shot search.

The orchestrator is currently powered by an open-source LLM hosted on LinkedIn’s internal infrastructure. It handles planning, memory tool calls, and final response generation within one unified reasoning loop. This simplifies the system design and keeps retrieval reasoning tightly integrated with answer synthesis. As user preferences and their tasks evolve, we are aiming to achieve adaptive memory layers which dynamically adapt ingestion and retrieval to new patterns of user behavior and task execution. In the short-term, we are improving the performance of orchestrators via SFT and RL training. We will leverage lessons learnt from domain-adapted foundation models to build a foundational, multi-turn instruction-tuned, scalable retrieval layer.

By elevating retrieval to a reasoning layer, CMA gains flexibility and robustness. Memory layers can evolve independently without changing application logic. Simple queries execute quickly with shallow plans, while complex, longitudinal queries trigger deeper orchestration. Most importantly, the system mirrors human memory access patterns, recalling facts, revisiting events, checking current context, and reusing prior strategies before forming a conclusion.

Access control

In building robust memory systems, proper setup of memory WRITE/READ permissions is crucial. This ensures effective memory sharing while preventing data leaks. Here’s how we manage access control across our system:

  • Isolated stores for each application: Each application operates within its isolated memory store, ensuring that data specific to an application remains confined and protected. This isolation is essential for maintaining security boundaries and minimizing unintended data exposure.
  • Ownership metadata tagging: Every piece of memory comes with ownership metadata. This tagging enables precise tracking of who owns each memory record, facilitating controlled access and strengthening data governance by associating records with specific users or entities.
  • User session restriction: Memory access is strictly limited to the logged-in user. This restriction helps ensure that data access is secure and confined to authorized user contexts. It prevents unauthorized data exposure outside the designated session.
  • No data sharing across tenants/users: Data sharing restrictions are strictly enforced, ensuring no unauthorized access across different tenants and users (In the Hiring Assistant for example, the memory is per recruiter and per customer tenant). This is vital for maintaining privacy and data integrity, especially in multi-tenant environments.

Evaluations

Memory can make an agent helpful over time, but it also has the potential to introduce user friction, higher cost and latency if persisted information in memory layers is incorrect or if the orchestrator returns hallucinated, incorrect and stale facts.

To help us evaluate the quality of an end-to-end multi-agent interaction with an end user, we use a three-tiered quality evaluation framework. It measures memory representation correctness, query–context–answer alignment, and task-success rate. The following explains this framework’s metrics in more detail:

  • Tier-1 evaluation metrics (continuous 0 to 1 scale). Tier-1 metrics quantify memory representation quality, which in turn measures the ingestion process correctness.  
    • We measure correctness by checking that the facts a memory claims about an entity (e.g., user or any other entity in the conversation) are supported by the underlying conversations and activity signals. We measure coverage by estimating how much of the anchored information in the source (entities and key attributes that should be remembered) is actually captured and identifiable in the generated memory.
  • Tier-2 evaluation metrics (continuous 0 to 1 scale). Tier-2 metrics measure CMA’s response quality using llm-as-judge. 
    • These metrics measure if CMA’s orchestrator retrieves information that is relevant to the user’s query, whether the response is grounded in the retrieved memories and not hallucinated, and if the final answer is complete to answer the user's query. 
    • This tier directly evaluates the full loop (i.e., query → retrieval plan across memory layers → selected memories → response) so we can pinpoint whether issues come from retrieval/routing and synthesis at serving time, as opposed to the underlying memory content written during ingestion.
  • Tier-3 evaluation metrics (% of goals successfully achieved). Tier-3 metrics evaluate end-to-end product performance, through offline helpfulness metrics, user-satisfactions studies and online A/B tests. 
    • We specifically look for improvement in (sub) task success rate and reduction in user friction.

Additionally, we also track system constraints to ensure the system scales and remains responsive. Beyond quality, we track system metrics because memory introduces real production constraints, such as:

  • Cost: Ingestion can require substantial LLM compute to extract, summarize, and structure long-term signals. 
  • Latency: Querying is latency sensitive, since orchestration should return a response under acceptable SLA.

Specifically, we monitor response latency, number of reasoning tokens and planning steps generated by the orchestrator and lastly token usage and LLM calls during ingestion.

Application integration

Applications interact with CMA via a natural language interface. Onboarding an application to leverage CMA, requires these steps:

  • Provision dedicated storage: Ensure isolated memory access by setting up dedicated storage.
  • Provide injectable business context, tool descriptions and hooks: CMA tools are generic, customizations to specific applications are provided via business context and detailed descriptions of memory tools. This information along with tool signature is passed as context to the CMA orchestrator. Optionally, the application team can provide custom embeddings or fine-tuned models to override the default ones, for EBR. This improves domain specific retrieval performance.
  • Ingest user activity data: The application starts ingesting user activity data and conversations through the CMA ingestion API.
  • Query through retrieval API: The application agent can query CMA using retrieval API and pass natural language query and relevant meta-data as arguments.

Example application: Hiring Assistant

Hiring Assistant is an agent designed for recruiters and powered by LinkedIn's extensive talent network. It assists recruiters in discovering and engaging candidates efficiently and at scale. Using CMA, the Hiring Assistant personalizes interactions with recruiters by leveraging past hiring projects and individual recruiter preferences.

Use case #1: Hiring Assistant leverages CMA to suggest the roles the recruiter could hire for

When a recruiter starts a hiring project, CMA is leveraged by the Hiring Assistant to fetch the recruiter's past projects and preferences to suggest a role or title that they can start hiring for. Typically recruiters hire for similar roles in a company, in such cases CMA provides them productivity boost and reduces user friction, as recruiters don’t need to enter their hiring project details again.

Use case #2: Augment hiring requirements 

Similar to the previous use case, preference derived from memory using CMA helps recruiters auto-populate information for a new project once they create a role for which they are hiring. In this case, past preferences from similar roles are leveraged to populate hiring requirements.

Use case #3: Retrieve and gain insights from past activities

Through the hiring assistant, the recruiter can get insights about their past hiring projects and activities on those projects. The Hiring Assistant internally queries CMA to obtain those facts and compiled insights.

Roadmap

As we continue to evolve CMA, our focus is on making memory more transparent, more powerful at reasoning time, and more robust as a foundational agentic system. Below are areas we’re investing in:

  • Transparent, user-controlled memory: Improve transparency, correctness, and trust by providing users control to inspect, edit and delete agent memory. This will shift memory from hidden optimization to a shared human–agent collaboration layer.
  • Dynamic test-time compute (deep thinking): Improve latency via cost-aware compute, by dynamically allocating reasoning depth based on query complexity.
  • Conflict detection & resolution: Improve relevance and recency of answers returned by CMA by identifying stale or contradictory memory entries, applying time based prioritization policy for memory sources, implementing conflict resolution strategies during ingestion and retrieval, and periodically purge and update memory.
  • Long-horizon simulation & benchmarking: Enhance our evaluation coverage and robustness by simulating and replaying multi-session interactions periodically to capture distribution shifts in user interactions, expand scope of online and offline metrics to measure personalization gains, product impact, graceful degradation, and end-to-end memory system performance.
  • Distributed filesystem abstraction: Enhance retrieval accuracy, memory storage scalability by leveraging filesystem mental model (hierarchy, versioning, lifecycle control). This distributed, memory native substrate will support scalable, high-frequency, semantically aware agent workloads.

Final thoughts

Agent intelligence does not emerge from models alone. It emerges from thoughtful harnesses built around the models.

Large language models provide powerful reasoning primitives but without memory, they remain stateless. Without domain grounding, they lack context. Without infrastructure, they cannot improve over time. Delivering a truly agentic experience requires weaving these pieces together into a cohesive, durable substrate.

CMA represents our first major step toward that: A horizontal memory platform designed to power stateful, context-aware, and deeply personalized agents at scale.

We’re still early in this journey. But we believe the future of AI agents will be defined not just by how well they think in the moment but by how well they remember, adapt, and grow with the people and organizations they serve. We’ll be sharing more info as we learn in future blog posts. Stay tuned!
