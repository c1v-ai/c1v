## Architecture Direction (Option A — LLM-Powered)

The approach is to build a **codebase-ingestion-agent** that:
1. Reads repo via GitHub API (file tree + key files)
2. Converts code signals into the existing `ExtractionResult` format
3. Feeds the existing Steps 1-6 artifact generation pipeline
4. Reuses all existing Zod schemas, specialist agents, and graph nodes

No AST parsing — LLM does the heavy lifting. Simpler to ship, language-agnostic.

## Code-to-SE Artifact Mapping (proven by THG)

| Code Signal | Detection Method | SE Artifact |
|---|---|---|
| Top-level module dirs (`apps/auth`, `apps/pilot_V2`) | File tree scan | Subsystem definitions |
| Route handlers (FastAPI `@app.get/post`, Next.js `route.ts`) | Pattern match in key files | Interface specs (IF-01..N) |
| ORM models, Pydantic/Zod schemas | Class/object detection | Data entities, N2 data payloads |
| External API calls (`fetch`, `requests.get`, `httpx`) | Import/call pattern match | External entities in context diagram + DFD |
| Function call chains | Cross-file reference analysis | Data flow sequences |
| Auth middleware, RBAC patterns | Pattern match | Cross-cutting concerns |
| `requirements.txt` / `package.json` | Dependency list parse | Technology constraints |
